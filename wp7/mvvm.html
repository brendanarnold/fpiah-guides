Silverlight application structure (MVVM)

Intro

  • MVVM stands for Model, View, View-Model
  • It is a 'pattern' i.e. a coding convention that is not enforced but has benefits if you follow it
  • You should follow this convention unless you have a good reason not to
  • The MVVM pattern tells you how to split up and organise you code for your application
  • It defines three roles for your code, the Model, the View and the View-Model
  • We usually create a directory in our project for each of these roles. Files are orgaised into these directories depending on their role

View

  • The View is code that deals with the UI
  • Pages and UserControls (i.e. custom widgets) defined in XAML and their 'code behind files' (i.e. the .cs or .vb files that are linked to the .xaml file in the Visual Studio project explorer) belong in the view
  • In the code-behind files we generally put only code to handle UI behaviour (e.g. showing a MessageBox or handling a button click) and not code that, for example, saves an object to a database

Model

  • The Model is the code that stores the actual data
  • Code that retrieves the data from the data source e.g. the webserver, database, XML files etc. belongs in the model.
  • The classes that contain the data from the datasource e.g. the 'Customer' class and the 'Invoices' class, also belong in the source

View-Model

  • The View-Model is a single class
